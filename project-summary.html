MasterFlow prototype direction

One simple front door for support requests and operational freight decisions

MasterFlow preserves existing business-team queues and request requirements, but removes the need for employees to hunt through a large ServiceNow catalog. It also gives Sales and Logistics one controlled workspace for freight opportunities that require human action.

The core ticketing experience

AI assembles the correct existing form; it does not eliminate required business information.

1. DescribeEmployee states the need in natural language.
2. MatchAI suggests an existing catalog request and confidence.
3. BuildKnown profile and message details are pre-filled.
4. ConfirmEmployee supplies only missing required fields.
5. RouteThe request enters the existing team queue with an SLA.

Locked product decisions

These decisions should not be casually reversed during the one-week build.

  • Every employee can create requests; receiver and administrator functions are role-controlled.
  • The home page stays minimal: one request input, P1 fast lane, My Requests, and Help Articles.
  • Existing catalog routing to team queues is preserved. MasterFlow improves intake rather than replacing ownership.
  • Different catalog items keep different required fields through one configurable dynamic-form engine.
  • Low-confidence or unknown requests route to Megan Delia - Triage in Business Enablement.
  • Shipping-stopped incidents bypass AI and create a P1 immediately.
  • Common template, routing, approval, role, and SLA changes should not require specialized development.
  • Production permissions would come from SSO; the role switch is for demonstration only.

Freight optimization operating model

Scan all eligible orders silently; create human work only when a decision, approval, investigation, or customer conversation is required.

Below 70% confidence

Normally record for analysis without interrupting operations. A high-value exception may be shown for human review.

70% to 89%

Create a review item only when expected savings exceed the economic threshold and the decision can happen before cutoff.

90% or higher

Potential automatic path only when customer promise, inventory, export, service, reliability, and approval guardrails all pass.

!
No indefinite operational holds.

A hold needs a named owner, decision deadline, and automatic release to the original approved shipping plan. Promise dates are never silently changed.

Required judging flow

The prototype must work end to end without live company integrations.

  1. Create a natural-language printer request, review the dynamic form, submit it, and see the numbered record in My Requests and the receiving queue.
  2. Use Shipping is stopped to create a P1 without relying on AI classification, then show the incident in the operational queue.
  3. Open freight opportunity ABI173, explain the 67% confidence and failed guardrails, and record a controlled decision.
  4. Open Request Template Manager and change a queue, SLA, trigger phrase, or required field without editing code.

Prototype boundaries

These integrations are represented honestly rather than claimed as live.

  • Fictional or sanitized demonstration data only.
  • Browser localStorage simulates persistence; there is no production database.
  • AI classification is a deterministic prototype matcher, ready to be replaced by an approved model.
  • ERP, OMS, OnlineComponents.com, Outlook, carrier, and SSO connections are simulated.
  • No customer email is sent and no live order hold is written to an operational system.
  • Migration, security hardening, model governance, and production audit controls are future implementation work.
