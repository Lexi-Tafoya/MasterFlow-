# Overdrive build rules

## What can be built

Build an application, agent, model, dashboard, automation, or tool that solves a real Master Electronics business problem. The team may use an approved problem or bring its own real ME problem. Scope may be narrowed, but the selected problem cannot be switched after Day 1.

The event is AI-first. Claude Code, Codex, Copilot, Cursor, AWS Bedrock, Databricks, or similar tools should be the primary build mechanism. Any language, framework, or platform is allowed, but the current MasterFlow prototype intentionally uses dependency-free HTML, CSS, and JavaScript for speed.

## Required constraints

- A working end-to-end prototype is required; slides alone do not qualify.
- The substantive build must occur during the official Build Period.
- Open-source libraries and approved internal shared code are allowed.
- Third-party tools are allowed only when their licenses permit internal use and sensitive data does not leave the company perimeter.
- Do not commit secrets, API keys, credentials, or sensitive company information to GitHub.
- Use approved secret-management practices if a later integration needs credentials.
- The tool budget is $500 per team for tools, API credits, or SaaS trials. Keep receipts and use the normal reimbursement process.

## MasterFlow implications

- Use fictional demo data only.
- Keep the prototype local and dependency-free unless a later decision justifies a paid or hosted service.
- Clearly label simulated AI, email, ERP, carrier, and order-hold behavior.
- Prioritize three working demo flows over incomplete production integrations.
